# Техническое задание: бот провайдера в другом мессенджере

> Документ описывает создание аналога текущего Telegram-бота в другом мессенджере (далее — **Target Messenger**), с сохранением бизнес-логики, сценариев и интеграций.

---

## 1. Цели и результат

### 1.1. Цель
Реализовать бота самообслуживания абонентов провайдера в Target Messenger с функционалом не ниже текущего Telegram-решения.

### 1.2. Результат
Готовый production-ready сервис, который:
- авторизует абонента по договору и паролю статистики;
- показывает баланс, тариф, платежи, новости;
- позволяет оформить обещанный платеж;
- формирует ссылку на оплату;
- поддерживает диалог с оператором;
- принимает push-уведомления от биллинга (персональные и массовые);
- имеет аудит, логирование, мониторинг, отказоустойчивость.

---

## 2. Термины
- **Абонент** — конечный пользователь бота.
- **Оператор** — сотрудник техподдержки.
- **Договор** — лицевой счет/контракт абонента в биллинге.
- **Billing API** — внешний API BGBilling (стандартный и кастомный).
- **Adapter Layer** — слой абстракции платформы мессенджера (получение/отправка сообщений, кнопки, callback).

---

## 3. Границы системы

### 3.1 Входит в scope
- Backend бота (Python, async).
- Интеграция с Target Messenger Bot API.
- Интеграция с Billing API.
- БД (SQLite на MVP / PostgreSQL для production).
- HTTP endpoints для уведомлений от биллинга.
- Логирование, метрики, healthcheck, админ-команды.

### 3.2 Не входит в scope
- Изменения в Billing API.
- Frontend-кабинет абонента.
- CRM/ServiceDesk (кроме нужной интеграции для оператора, если отдельно не оговорено).

---

## 4. Функциональные требования

## 4.1 Онбординг и главное меню
1. По стартовой команде бот приветствует пользователя.
2. Если пользователь ранее авторизован — показывает персонализированное меню.
3. Если не авторизован — предлагает авторизацию и связь с поддержкой.
4. Для специального операторского аккаунта выводится служебное меню.

## 4.2 Авторизация абонента

### Сценарий
1. Пользователь запускает авторизацию.
2. Бот запрашивает номер договора.
3. Валидирует: только цифры, длина 3–6.
4. Бот запрашивает пароль статистики.
5. Вызывает `login` Billing API.
6. При успехе сохраняет привязку `messenger_user_id ↔ contract_id`.
7. При неуспехе показывает корректную ошибку без утечки чувствительных деталей.

### Требования
- FSM/стейт-машина обязательна.
- Должна быть кнопка отмены на каждом шаге.
- В логи нельзя писать пароль.

## 4.3 Баланс
- Доступно только авторизованным.
- Вызов `/jsonWebApi/contractBalance`.
- Показ суммы и валюты.

## 4.4 Тариф
- Доступно только авторизованным.
- Вызов `/jsonWebApi/contractTarifPlans`.
- Показ названия тарифа и периода действия.
- Дополнительно: вызов `recommendPayment` и показ рекомендуемой суммы.

## 4.5 Последние платежи
- Доступно только авторизованным.
- Вызов `/jsonWebApi/lastContractPayments`.
- Отображать минимум 3 последних платежа (дата/сумма/тип).

## 4.6 Новости
- Доступно только авторизованным.
- Вызов `/jsonWebApi/newsList`.
- Очистка HTML перед отправкой.
- Сортировка по дате и вывод последних 3 новостей.

## 4.7 Обещанный платеж
- Доступно только авторизованным.
- Подтверждение действия перед API-вызовом.
- Вызов `/api/rest/0/telegramApi/lowerLimit` (endpoint может быть переименован при портировании, контракт обязателен).
- Показ результата (`newLimit`, `days`) или причины отказа.

## 4.8 Оплата услуг
- Доступно только авторизованным.
- Запрос суммы (минимум 20 рублей).
- Валидация формата числа.
- Формирование URL оплаты с параметрами: service/shop_id, договор, amount в копейках.
- Кнопка перехода к оплате + кнопка возврата.

## 4.9 Техподдержка (чат абонент ↔ оператор)

### Сценарий абонента
1. Включает режим поддержки.
2. Отправляет текст/медиа.
3. Бот пересылает оператору с мета-информацией (имя, username, договор).
4. Получает подтверждение доставки.

### Сценарий оператора
1. Оператор отвечает reply на пересланное сообщение.
2. Бот находит связку по `support_message_id`.
3. Пересылает ответ абоненту как reply на исходное сообщение.

### Ограничения
- Голосовые можно запретить (как в текущей реализации) или реализовать транскрибацию — по согласованию.
- Не должно быть «потери контекста» reply-цепочек.

## 4.10 Уведомления от биллинга

### Endpoint A: персональное уведомление
`POST /billing/notify`

Request:
```json
{
  "contract_id": "703",
  "message": "Ваш баланс изменился"
}
```

Behavior:
- Найти все привязанные `messenger_user_id` по `contract_id`.
- Разослать сообщение каждому.
- Вернуть статус `ok|partial|error` с counters `sent/failed`.

### Endpoint B: массовая рассылка
`POST /billing/broadcast`

Request:
```json
{
  "message": "Плановые работы в 02:00"
}
```

Behavior:
- Взять всех пользователей из БД.
- Разослать сообщение всем.
- Вернуть `sent/failed`.

## 5. Нефункциональные требования

## 5.1 Надежность
- Сервис должен выдерживать перезапуски без потери критичных данных.
- Все внешние запросы с таймаутами и обработкой исключений.
- Идемпотентность входящих webhook (по возможности через request-id).

## 5.2 Производительность
- Время ответа бота пользователю: P95 ≤ 2 сек при нормальном состоянии Billing API.
- Асинхронный стек обязателен.

## 5.3 Безопасность
- Обязательная проверка Bearer-токена на billing endpoints.
- Секреты только в env/secret store.
- Не логировать пароли и чувствительные данные.
- Если пароль нужен для повторных запросов в биллинг: хранить только в шифрованном виде (reversible encryption, например Fernet/AES-GCM), ключ — в secret-store/KMS.
- TLS для внешних endpoint.
- IP allowlist для billing-вызовов (если доступно на периметре).

## 5.4 Наблюдаемость
- Структурные логи с correlation id.
- Метрики: количество сообщений, ошибки API, latency, активные support sessions.
- Алерты: деградация Billing API, всплеск ошибок, падение процесса.

## 6. Архитектурные требования

## 6.1 Слои
1. **Transport/Adapter (Target Messenger SDK/API)**
2. **Handlers / Application Use Cases**
3. **Billing Client**
4. **Repository (DB layer)**
5. **Infrastructure (logging, config, health, metrics)**

## 6.2 Совместимость бизнес-логики
- Все пользовательские сценарии должны быть функционально эквивалентны текущему Telegram-боту.
- UI-компоненты (кнопки, inline/reply) адаптируются под возможности Target Messenger.

## 6.3 Хранилище данных
Минимальные сущности:
- `users(id, messenger_user_id, contract_id, contract_title, created_at, updated_at)`
- `support_sessions(messenger_user_id, contract_title, is_active, updated_at)`
- `support_messages(id, messenger_user_id, user_msg_id, operator_msg_id, created_at)`

**Важно:** если бизнес-процесс требует хранить `password`, хранить только шифротекст + ротация ключа + план миграции/ре-шифрования.

---

## 7. API-контракты и нормализации

## 7.1 Нормализация ответа login
Если Billing API может вернуть разные ключи, в доменной модели строго использовать:
- `contract_id: str`
- `contract_title: str`
- `status: str`

## 7.2 Нормализация новостей
- HTML → plain text.
- Сохранение абзацев/переносов.
- Ограничение длины одного сообщения (если мессенджер ограничивает размер).

## 7.3 Валидации
- `contract_id` input: `^\d{3,6}$`
- `amount`: decimal > 0, min 20.00 RUB
- Billing webhook payload: обязательные поля + типы

## 8. Админ-функции
- Команда/endpoint проверки статуса сервиса.
- Команда мягкой остановки/рестарта (по role-based ACL).
- Логирование admin-действий в audit trail.

## 9. Роли и доступы
- **Абонент**: пользовательские функции (без админ-команд).
- **Оператор**: ответы в поддержку, просмотр входящих обращений.
- **Администратор**: служебные команды и диагностика.

Требуется конфигурируемый список ID администраторов и операторов.

## 11. План реализации (этапы)

### Этап 1 — Foundation (1–2 недели)
- Каркас проекта, config, logger, DB schema, healthcheck.
- Adapter для Target Messenger (получение/отправка сообщений).

### Этап 2 — Базовый функционал (1–2 недели)
- start/menu, auth FSM, balance/tariff/payments/news.
- Billing client + нормализации.

### Этап 3 — Поддержка и платежи (1 неделя)
- support sessions + reply mapping.
- promised payment + payment URL.

### Этап 4 — Интеграции и hardening (1 неделя)
- billing webhooks + auth.
- ACL admin-команд, retry policy, observability.

### Этап 5 — Тесты и запуск (1 неделя)
- интеграционные тесты,
- нагрузочный smoke,
- acceptance с бизнесом,
- production rollout.

## 12. Тестирование и критерии приемки

## 12.1 Функциональные тест-кейсы (минимум)
1. Успешная авторизация валидного договора.
2. Ошибка авторизации при неверном пароле.
3. Неавторизованный не может получить баланс/тариф/платежи/новости.
4. Новости корректно очищаются от HTML.
5. Обещанный платеж: success и business-error сценарии.
6. Поддержка: сообщение абонента доходит оператору, ответ оператора reply возвращается абоненту.
7. Webhook notify: корректная доставка по contract_id.
8. Webhook broadcast: массовая рассылка с корректной статистикой.
9. Проверка ACL админ-команд.
10. Отмена любого FSM-сценария работает корректно.

## 12.2 Нефункциональные
- Все API-вызовы имеют timeout.
- Нет хранения пароля в открытом виде.
- Логи не содержат секретов.
- Ошибки внешних сервисов не приводят к падению процесса.

## 13. Риски и ограничения
- Ограничения Target Messenger по типам сообщений/кнопкам/reply-механике.

Митигировать через adapter-абстракцию, retry, очереди (при необходимости), и интеграционные тесты.
